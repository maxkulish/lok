Risks & Known Issues

Risk assessment and recommendations for lokomotiv v20260208.0.2

Identified Risks

Workflow Shell Execution

Medium

Workflow TOML files can define arbitrary shell commands via the shell field. While this is by design (user-authored pipelines), it means any TOML workflow file should be treated as executable code.

Evidence: src/workflow.rs:2041-2048 - Executes sh -c <command> for shell steps.
Mitigation: Only run workflows from trusted sources. Review .lok/workflows/ content before execution. The application does not download workflows from the internet.

Code Edit Application

Medium

The implement command and workflows with apply_edits = true can modify source files based on LLM output. While checkpoints provide rollback, malformed edits could corrupt files.

Evidence: src/tasks/implement.rs:454, src/workflow.rs:2260 - Write files from LLM-generated content.
Mitigation: Git checkpoints created before edits. Use --dry-run where available. Review changes before committing. The verify step can validate edits.

LLM Prompt Injection via Source Code

Low

When analyzing codebases, source file contents are included in prompts. Maliciously crafted source files could attempt prompt injection on the LLM backends.

Evidence: Codebase context is prepended to prompts in src/tasks/mod.rs and src/context.rs.
Mitigation: This is inherent to any LLM-based code analysis tool. The risk is limited because lok itself does not auto-execute LLM responses (except when explicitly told to via implement/workflow). Use read-only mode (codex backend) for untrusted code.

API Key Exposure in Logs

Low

While API keys use SecretString, the --verbose flag shows additional debug output. Care must be taken that verbose logging does not inadvertently expose secrets.

Evidence: src/backend/claude.rs:59 - API key loaded as SecretString, only exposed via .expose_secret() for HTTP headers.
Mitigation: SecretString prevents accidental Display/Debug printing. The .expose_secret() call is only used in the HTTP header construction, not in logging paths.

Cache Poisoning

Low

Cache files at ~/.cache/lok/ contain LLM responses in JSON. If an attacker gains write access to this directory, they could inject malicious cached responses.

Evidence: src/cache.rs:300 - Cache stored as JSON files keyed by SHA256 hash.
Mitigation: Cache directory permissions follow user defaults. Disable cache with --no-cache flag or cache.enabled = false in config if concerned.

Recommendations

  • 1
    Review workflow files before execution. Treat .lok/workflows/*.toml as executable code. Never run workflows from untrusted sources without review.
  • 2
    Use --dry-run for fix/implement commands on unfamiliar codebases to preview changes before application.
  • 3
    Set API keys via environment variables rather than config files. The codebase already supports this pattern.
  • 4
    Use the Codex backend in read-only mode (-s read-only, already the default) when analyzing untrusted repositories.
  • 5
    Review git checkpoints after implement/fix operations. The git-agent creates rollback points but they require manual verification.
  • 6
    Be aware of prompt content. When querying LLMs about a codebase, relevant source code is included in prompts. Ensure this is acceptable for your API provider's data policies.
