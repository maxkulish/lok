Architecture & Data Flow

Module structure, data flow, and storage for lokomotiv v20260208.0.2

Data Flow

  User CLI Input (e.g., "lok ask 'find bugs in this code'")
       |
       v
  [main.rs] Parse CLI args (clap derive)
       |
       v
  [config.rs] Load lok.toml
       |  1. Explicit --config path
       |  2. ./lok.toml (project)
       |  3. ~/.config/lok/lok.toml (user)
       |  4. Built-in defaults
       v
  [backend/mod.rs] Initialize enabled backends
       |
       +--- Check availability (which, env vars, network)
       |
       v
  Route by command type:
       |
       +---> ask/hunt/audit/fix/ci
       |      |
       |      +---> [cache.rs] Check cache (SHA256 of prompt+backends+cwd)
       |      |       |
       |      |       +--- HIT: Return cached results
       |      |       +--- MISS: Continue to query
       |      |
       |      +---> [tasks/mod.rs] Inject context (N+1, security patterns)
       |      |
       |      +---> [backend/mod.rs] run_query()
       |      |       |
       |      |       +--- Parallel: tokio::join_all(backends)
       |      |       +--- Sequential: iterate one by one
       |      |       |
       |      |       +--- Each backend:
       |      |       |     Codex  --> subprocess: codex exec --json
       |      |       |     Gemini --> subprocess: npx @google/gemini-cli
       |      |       |     Claude --> reqwest POST api.anthropic.com
       |      |       |              or subprocess: claude -p
       |      |       |     Ollama --> reqwest POST localhost:11434
       |      |       |     Bedrock -> AWS SDK InvokeModel
       |      |       |
       |      |       +--- Collect QueryResult { backend, output, success, elapsed_ms }
       |      |
       |      +---> [cache.rs] Store results (TTL: 24h default)
       |      |
       |      +---> [output.rs] Format and display
       |
       +---> conduct
       |      +---> [conductor.rs] Multi-round Claude orchestration
       |             Tool-use loop: Claude picks backends, refines results
       |
       +---> debate
       |      +---> [debate.rs] Multi-backend debate with synthesis
       |             Rounds: initial -> rebuttals -> consensus
       |
       +---> spawn
       |      +---> [spawn.rs] Break into parallel agent subtasks
       |             Parse AGENT: format, execute in parallel, synthesize
       |
       +---> workflow
       |      +---> [workflow.rs] TOML pipeline engine
       |             Dependency resolution -> parallel execution -> interpolation
       |             Supports: loops, conditions, edits, verification
       |
       +---> implement
              +---> [tasks/implement.rs] ARF spec execution
                     Step-by-step with checkpoints and rollback

Module Structure

src/main.rs

2,030 lines

CLI entry point. Defines all commands via clap derive. Routes to appropriate handlers. Contains PR review, diff analysis, doctor check, and init logic.

src/workflow.rs

3,988 lines

Workflow execution engine. TOML-based pipelines with dependency resolution, parallel execution, interpolation ({{ steps.X.output }}), conditions, loops, edit application, and verification.

src/backend/mod.rs

300 lines

Backend trait definition and factory. Parallel/sequential query runner with per-backend timeouts and progress tracking.

src/backend/claude.rs

215 lines

Dual-mode Claude backend: API mode (reqwest to Anthropic) or CLI mode (claude subprocess). SecretString for API key protection.

src/backend/codex.rs

89 lines

Codex CLI wrapper. Spawns codex exec in read-only mode, parses JSON event stream for agent messages.

src/backend/gemini.rs

83 lines

Gemini CLI wrapper via npx. Configurable output line skipping. 600s default timeout for agentic tasks.

src/backend/ollama.rs

111 lines

Ollama HTTP client. REST API to local server. Default model: llama3.2. Configurable base URL.

src/backend/bedrock.rs

159 lines

AWS Bedrock integration (optional feature). Uses AWS SDK with standard credential chain.

src/config.rs

464 lines

TOML configuration parsing. Default backend configs, task definitions, and config file discovery (project -> user -> defaults).

src/cache.rs

403 lines

Query result caching. SHA256 key generation, TTL-based expiration, graceful degradation on errors.

src/conductor.rs

317 lines

Multi-round Claude API orchestration. Tool-use protocol for delegating to backends. Configurable max rounds and tokens.

src/context.rs

421 lines

Codebase context detection. Identifies languages, frameworks, infrastructure, CI/CD via file scanning.

src/delegation.rs

355 lines

Task classification and backend recommendation. Maps task types to backend strengths for optimal routing.

src/debate.rs

313 lines

Multi-backend debate engine. Backends respond, critique each other's answers, then synthesize consensus.

src/spawn.rs

378 lines

Parallel agent spawning. Breaks tasks into subtasks via Claude, executes in parallel, synthesizes results.

src/git_agent.rs

513 lines

Git checkpoint and event tracking. Stores agent events on orphan branch. Supports rollback on failure.

src/tasks/ (7 files)

~2,100 lines total

Task implementations: hunt (bug detection), implement (ARF specs), spec (spec generation), fix (issue fixing), ci (CI analysis), context (codebase scanning), audit (security).

src/consensus.rs

305 lines

Consensus algorithms and voting for multi-backend agreement. Used by debate and team modes.

What It Does NOT Do

  • No GUI or web server - CLI only, no HTTP listener
  • No autonomous decisions - runs queries, returns results
  • No file watching or auto-rerun
  • No secrets vault or encrypted storage
  • No database - all state is file-based
  • No real-time streaming output during execution
  • No auto-install of backend CLIs
  • No model training or fine-tuning
  • No API usage/cost tracking
  • No multi-turn conversation history between invocations
