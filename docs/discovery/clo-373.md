# Discovery: CLO-373

## Problem framing

Lok backend/parser contributors are affected because the upcoming Codex JSONL parser (FR-3a) has no stable checked-in corpus to test against. Today `src/backend/codex.rs` parses Codex output by scanning JSONL text for `item.completed` lines containing `agent_message`, but there are no fixtures covering success, failure, reasoning usage, or missing-agent-message edge cases. We want `tests/fixtures/codex/` to contain scrubbed, real `codex exec --json` streams plus capture documentation so parser work can be written test-first. This is needed now because the phase-2 PRD explicitly makes FR-40 a prerequisite before opening the FR-3a parser PR.

## PRD source

The task is already covered by `docs/prds/prd-phase-2-predictable-cli-execution-v5.md`:

- §4, FR-40 requires `tests/fixtures/codex/` with successful `turn.completed`, `turn.failed`, multi-turn/reasoning, and missing-`agent_message` streams.
- §11.4 requires capturing real streams before writing the event-driven parser and confirms the event set to preserve: `thread.started`, `turn.started`, `turn.completed`, `turn.failed`, `item.started`, `item.completed`, and `error`.
- The Linear issue includes a PRD/source section and acceptance criteria, so no per-issue PRD file is needed.

## Existing code

### Current Codex parser

`src/backend/codex.rs` currently keeps parsing intentionally simple:

- Default args are `codex exec --json --ephemeral` plus dynamic sandbox/model injection.
- `parse_output()` iterates stdout lines and returns the first `item.text` from a line that contains both `"type":"item.completed"` and `agent_message`.
- The fallback returns raw stdout if no matching agent message is found.

This is exactly the brittle path FR-3a plans to replace. The fixtures in this task should therefore preserve enough event variation to pin future parser behavior, not change parser code yet.

### Existing fixtures/tests layout

- `tests/` currently contains workflow TOML fixtures only; there is no `tests/fixtures/` directory yet.
- Grep found no existing fixture loader conventions to follow for backend event streams.
- Codex JSONL examples exist only in review artifacts such as `docs/reviews/CLO-374-codex-validation.md`; these are useful as shape references but are not the requested checked-in test corpus.

### Reference docs

- `docs/investigations/codex-quick-ref.md` documents the relevant Codex flags and event types.
- Local Codex version check reports `codex-cli 0.130.0`, which satisfies the issue requirement of `>= v0.119.0` and supports `--json`/`--ephemeral`.

## Baseline score

**8 / 10** — The work is additive and isolated to `tests/fixtures/codex/` plus a README. The main uncertainty is not code architecture; it is reliably producing the two unusual real streams (`reasoning_output_tokens > 0` and completed-without-final-`agent_message`) without leaking local paths/secrets.

## Approaches considered

### Approach A: Capture fresh real streams with targeted prompts (selected)

**Summary:** Use local `codex exec --json --ephemeral` runs for each scenario, choose prompts/sandbox conditions to exercise the requested events, then scrub and validate JSONL before committing.

**Pros:**
- Directly satisfies the Linear scope: fixtures are from real Codex output.
- Pins current Codex 0.130.0 event shapes before parser work begins.
- Keeps implementation small and reviewable.

**Cons:**
- Some scenarios may be prompt/model-dependent, especially reasoning tokens and missing final agent message.
- Requires manual scrubbing and validation discipline.
- Real streams may include environment-specific command text that must be normalized.

**Effort:** S
**Risk:** Low/medium

### Approach B: Hand-author minimal synthetic JSONL fixtures

**Summary:** Create concise JSONL files matching the documented event shapes without invoking Codex.

**Pros:**
- Fast, deterministic, and easy to shape for edge cases.
- Avoids secret/path leakage risk from captured streams.
- Fixtures can be very small.

**Cons:**
- Violates the issue requirement that streams are captured from real `codex exec --json` runs.
- Lower confidence that future parser tests match actual CLI output.
- Could miss real event ordering/field details.

**Effort:** S
**Risk:** Medium

### Approach C: Defer fixture capture into the FR-3a parser PR

**Summary:** Wait until parser tests are being written, then capture only the fixtures those tests need.

**Pros:**
- Parser author can tune fixture coverage to exact parser requirements.
- Avoids doing work before parser design settles.

**Cons:**
- Contradicts FR-40 sequencing; FR-40 exists specifically to land before FR-3a.
- Keeps parser work blocked on live Codex availability.
- Increases parser PR scope and review risk.

**Effort:** M later
**Risk:** High

## Chosen approach

**Approach A: Capture fresh real streams with targeted prompts.** It is the only approach that satisfies the explicit acceptance criteria while keeping this PR independent of parser implementation. The implementation should add the fixture directory, document exact capture/scrub steps, run a JSONL validator over every fixture, and record Codex version `0.130.0` for reproducibility.

## Implementation notes for next phases

- Create `tests/fixtures/codex/README.md` with:
  - exact command template: `codex exec --json --ephemeral "<prompt>" > tests/fixtures/codex/<scenario>.jsonl`
  - Codex version used
  - end-event check (`turn.completed` or `turn.failed`)
  - scrub checklist for secrets, API keys, absolute local paths, usernames, and project-private temp paths
  - JSONL validation command, e.g. `python3 - <<'PY' ... json.loads(line) ... PY`
- Add exactly the four requested fixture files:
  - `turn-completed.jsonl`
  - `turn-failed.jsonl`
  - `multi-turn-reasoning.jsonl`
  - `missing-agent-message.jsonl`
- Prefer prompts that keep outputs short and non-sensitive.
- If Codex includes local paths in tool-call or error output, replace them with neutral placeholders while preserving valid JSON strings and event structure.
- Do not modify `src/backend/codex.rs` in this task unless validation exposes an unavoidable fixture-format issue; parser changes belong to FR-3a.
