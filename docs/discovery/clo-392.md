# Discovery Report: CLO-392 ‚Äî FR-13: Codex health probe + version-aware unusable-flag matrix

## Problem Framing

Lok users who configure Codex backend steps with newer flags (e.g., `--output-schema`, `--ignore-user-config`) currently receive no upfront validation of whether their installed `codex` CLI supports those flags. The existing `CodexBackend::health_check` only verifies that the binary is present on PATH; when an unsupported flag is used at query time, the Codex CLI fails at runtime, producing a poor user experience.

This ticket fills in the Codex-specific health probe to detect the installed version, compare it against a canonical flag matrix, and surface unsupported flags in `HealthStatus.notes`/`unusable_flags` so the engine can warn during warmup.

## Existing Code

Key locations:

- `src/backend/codex.rs` ‚Äî `CodexBackend` struct and trivial `health_check` that only calls `which::which`.
- `src/backend/context.rs` ‚Äî `HealthStatus` already has `version`, `unusable_flags`, `models`, etc.
- `src/backend/mod.rs` ‚Äî `Engine::warmup_backends` parallelizes probes into `BACKEND_CACHE`; `is_available` is a cache-only lookup.
- `src/backend/ollama.rs` ‚Äî `OllamaBackend::health_check` demonstrates the active-probe pattern: HTTP request with 2 s timeout, JSON deserialization, fully populated `HealthStatus`.
- `docs/investigations/codex-quick-ref.md` ‚Äî Canonical flag ‚Üí version matrix already documented.

## Baseline Score: 8/10

No rewrite needed. The `Backend::health_check` trait surface, `HealthStatus` struct, and `BACKEND_CACHE` infrastructure are all in place from CLO-371 and CLO-388. The remaining work is contained to the Codex backend‚Äôs `health_check` body and associated unit tests.

## Approaches

### 1. In-place CodexBackend probe (Minimal)
Implement `codex --version` directly inside `CodexBackend::health_check`, parse semver, populate `HealthStatus.version` and `unusable_flags` from a `const` matrix table inside `src/backend/codex.rs`.

- **Pros**:
  - Simple, contained diff (~50‚Äì70 lines of core logic + tests).
  - Follows the exact pattern demonstrated by `ollama.rs`.
  - Aligns with the AC‚Äôs instruction to extract the matrix into `src/backend/codex.rs`.
- **Cons**:
  - Not reusable for other potential CLI backends.
- **Effort**: S
- **Risk**: Low

### 2. Generic `SubprocessBackendVersion` helper
Create a shared helper/module that probes `--version`, parses SemVer, and evaluates a per-backend flag matrix.

- **Pros**:
  - Reusable if future CLI backends need version probes.
- **Cons**:
  - Over-engineering today: only Codex needs this exact pattern.
  - May require refactoring `Backend` trait or adding a new sub-trait.
- **Effort**: M‚ÄìL
- **Risk**: Medium

## Chosen Approach: 1 ‚Äî In-place CodexBackend probe

The infrastructure is intentionally designed for per-backend probes (CLO-388 architecture). Keeping the probe logic local to `codex.rs` matches the established pattern and minimizes risk. A generic abstraction can be extracted later if a second CLI backend emerges that needs identical logic.

## Discovery Debt

None. All dependencies (CLO-371 trait, CLO-388 cache) are merged and stable. The flag matrix is already documented in `docs/investigations/codex-quick-ref.md`.
