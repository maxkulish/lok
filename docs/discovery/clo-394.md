# Discovery Report: CLO-394 — FR-12a Replace Gemini CLI backend with opencode subprocess

## 1. Problem framing

**WHO:** Lok users and automation pipelines that use the `gemini` backend for `lok ask`, `lok run`, or rs-wisper review/fix flows; this includes any environment still relying on Gemini CLI for model-backed backend calls.
**WHAT:** `src/backend/gemini.rs` still shells out to deprecated `@google/gemini-cli` via `npx`, builds a command string via `echo '' | npx ... '<prompt>'`, and currently parses output by line-skipping (`skip_lines`) while treating usage as best-effort text fallback.
**WHY NOW:** Google’s migration notice marks the Gemini CLI as deprecated, and this task must move execution to `opencode` with the documented `--model`, `--agent`, and positional prompt contract before the old CLI path becomes unsupported.

## 2. Existing code

### `src/backend/gemini.rs`
- `GeminiBackend::new()` defaults `command: Some("npx")`, args `[@google/gemini-cli, --output-format, json]`, and `skip_lines: 1`.
- `build_shell_cmd()` currently builds one shell string:
  - `sh -c` + `echo '' | ...` + quoted prompt + approval-mode mapping.
- `query()` executes via `Command::new("sh").arg("-c")`, captures stdout/stderr, parses either:
  - JSON envelope with `GeminiEnvelope` (`response`, optional `stats`) **or**
  - fallback text from `parse_output()`.
- `resolve_effective_sandbox()`/`approval` mapping uses legacy `--approval-mode plan/auto_edit/yolo`.
- Tests in `src/backend/gemini.rs` validate shell string composition and fallback parsing.

### Configuration and command checks
- `src/config.rs` sets default Gemini backend to:
  - `command = Some("npx")`
  - `args = ["@google/gemini-cli", "--output-format", "json"]`
  - `skip_lines = 1`
  - `test_gemini_backend_defaults` only asserts command and skip-lines.
- `src/main.rs` currently advertises doctor checks for (`gemini`, `npx`) and API key (`GOOGLE_API_KEY`).

### Tests and fixtures
- `tests/fixtures/gemini/*.json` + `tests/gemini_fixtures.rs` already assert CLI-style envelope validity and provide malformed/`error-envelope` coverage.
- Existing tests for usage parsing in `src/backend/gemini.rs` are tied to the old `stats.models.<model>.tokens` shape.

### Baseline score

**6/10** — the backend and fixture/tests scaffolding are present, so the work is implementable in one focused backend migration pass. But the current shell-string execution model (`sh -c`, quoted prompt, CLI shim assumptions) does not align well with opencode’s documented positional argument and JSON event semantics, so this is more than a pure config swap.

## 3. Approaches identified

### Approach A: Minimal command-string swap with tolerant parser update

Keep `GeminiBackend::query()` architecture and command-string construction, replacing the Gemini CLI tokens in default config/command strings with `opencode` equivalents.

- **Pros:** lowest diff; reuses most existing parse/test scaffolding.
- **Cons:** preserves shell-based quoting/escaping risks and makes stdin/stderr routing harder to evolve; relies on prompt being positional in a shell string.
- **Effort:** M
- **Risk:** Medium

### Approach B: Refactor Gemini backend to direct `tokio::process::Command` invocation (recommended)

Build an argv vector and call `Command::new("opencode")` directly (`run`, `--format`, `json`, `--model`, `--agent`, `--` prompt), then parse aggregated stdout (and/or stream events) into response + usage.

- **Pros:** avoids shell quoting pitfalls, aligns with existing Codex/Claude command patterns, easier to add future stdin/event parsing and robust tests.
- **Cons:** requires touching more code paths (`build_shell_cmd`, tests that assert raw shell string, prompt handling path).
- **Effort:** M
- **Risk:** Low–Medium

### Approach C: Use a streaming event parser only (pure event mode)

Invoke opencode in streaming mode and consume line-delimited JSON events to extract final message + usage counters, treating malformed streams as typed parse failures.

- **Pros:** maximally robust if opencode event stream is stable; supports richer diagnostics and future fields.
- **Cons:** highest complexity now; requires fixture/model assumptions that are not fully captured yet.
- **Effort:** L
- **Risk:** Medium

## 4. Chosen approach

**Approach B** — direct `Command` invocation with full argv mapping and tolerant JSON parse.

It directly addresses the deprecation migration with a safer execution model and does not encode behavior in a shell string. It also makes sandbox-to-`--agent` mapping and model injection explicit and keeps the backend extensible for opencode event-shape drift.

## 5. Discovery debt

None.

## 6. Implementation outline (high level)

- `src/backend/gemini.rs`
  - Replace default command shell construction with direct `opencode` argv.
  - Map `SandboxMode` to `--agent` flags, adding `--dangerously-skip-permissions` for `danger-full-access`.
  - Support model override as `--model google/<model>` and pass prompt as positional argument (or stdin fallback if needed).
  - Replace `GeminiEnvelope` parsing logic to opencode event shape while keeping `QueryOutput` population behavior (`stdout`, `stderr`, `usage`, `model`).
- `src/config.rs`
  - Update default Gemini backend command/args/skip-lines to `opencode`, `run --format json`, and longer timeout.
- `src/main.rs`
  - Doctor checks: validate `opencode` availability; remove strict `GOOGLE_API_KEY` requirement for gemini path.
- `src/backend/context.rs` comment
  - Replace outdated approval-mode mapping note with opencode agent routing note.
- Tests:
  - Update/extend existing Gemini unit tests for `build_shell_cmd`/command assertions and new usage extraction behavior.
  - Add/adjust fixtures if parser shape needs new fields.
