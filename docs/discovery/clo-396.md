# Discovery: CLO-396 — FR-12c: opencode migration docs + setup guide refresh

## Problem framing

**WHO:** Users, contributors, and automation pipelines that reference lok's setup
guide, README quickstart, or backend reference tables. When CLO-394 (FR-12a)
ships the opencode-backed `gemini` backend, every user-facing doc that mentions
`@google/gemini-cli`, `npx`, or `GEMINI_API_KEY`/`GOOGLE_API_KEY` becomes a
source of confusion — users will install the wrong tool, set wrong env vars, and
hit authentication failures that the docs don't explain.

**WHAT currently:** The `docs/guides/lok-setup-guide.md` Gemini backend section
assumes `npx @google/gemini-cli` with `GEMINI_API_KEY` as a required env var,
and its sandbox mapping references `--approval-mode` (a gemini-cli flag that
opencode doesn't use). The `README.md` quickstart example shows `gemini - ready`
in the `lok doctor` output, and its prerequisites table still lists `gemini-cli`
without mentioning `opencode auth login`. Backend reference tables
(`Token Usage Observability` in the setup guide) reference the old Gemini CLI
envelope shape.

**WHAT we want:** All user-facing docs updated to the opencode era:
- Install via `brew install anomalyco/tap/opencode` or
  `curl -fsSL https://opencode.ai/install | bash`
- Auth via `opencode auth login` (Google OAuth)
- Sandbox mapping: `read-only → --agent plan`, `workspace-write → --agent build`,
  `danger-full-access → --agent build --dangerously-skip-permissions`
- `GEMINI_API_KEY`/`GOOGLE_API_KEY` mentioned as optional fallback only
- A "Migrating from Gemini CLI" callout so existing users know the delta
- No remaining `@google/gemini-cli`, `npx` as required tool, or old env-var
  requirements

**WHY now:** CLO-394 is shipping the opencode backend switch in this release
window. The docs must land in the same release or immediately after — shipping
the code change without the doc update means every user who runs `lok doctor`
or reads the setup guide will see contradictory information. CLO-395 (opencode
health probe) also ships alongside, so the auth section needs to align with
its probe output.

## Approaches

### Approach A: Surgical in-place updates (recommended)

Update only the affected sections in each file. Replace the Gemini backend
content with opencode equivalents. Add a "Migrating from Gemini CLI" callout.
No new files, no structural changes.

**Pros:**
- Minimum diff — easy to review and merge
- Preserves all proven content (Claude, Codex, Ollama sections are unchanged)
- Matches the issue's well-scoped checklist

**Cons:**
- May miss stale references in less-obvious locations (cross-references in other docs)
- No standalone migration doc for users who want a before/after comparison

**Effort:** S (1-2 hours)
**Risk:** Low

### Approach B: Full doc audit + rewrite

Read every file in `docs/`, `docs/guides/`, `docs/investigations/`, and the
README to catch every stale reference. Rewrite affected sections
comprehensively. Possibly add a standalone `docs/guides/migrating-from-gemini-cli.md`.

**Pros:**
- Catches hidden stale references (investigations, design docs, specs)
- Migration doc gives users a clear before/after path

**Cons:**
- High effort for a small surface of changes
- Design docs and investigations reference deprecated tools *by design*
  (they document the old state); updating them would be misleading
- Risks introducing errors by touching content that doesn't need changing

**Effort:** M (3-5 hours)
**Risk:** Medium

### Approach C: Split — in-place updates + standalone migration doc

Same as Approach A, but also create a
`docs/guides/migrating-from-gemini-cli.md` with the install + auth + sandbox
delta, linked from the setup guide and README.

**Pros:**
- All the benefits of Approach A
- Migration doc serves as a clear upgrade path for existing users
- Can be independently versioned and linked from changelog/release notes

**Cons:**
- Extra file to maintain as the migration period ends
- The "Migrating from Gemini CLI" callout in the setup guide (per ACs) may
  be sufficient — a standalone doc may not add proportional value

**Effort:** M (2-3 hours)
**Risk:** Low

## Approach chosen: Approach A — Surgical in-place updates

The scope is well-defined by the issue ACs, and the CLO-394 design doc already
details the exact replacement values (opencode command, sandbox mapping, auth
flow). Writing a standalone migration doc is unnecessary for this task — the
setup guide callout covers the delta. If user confusion persists, Approach C
can be adopted as a follow-up.
