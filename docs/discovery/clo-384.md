# CLO-384 Discovery Report: FR-23 — Per-step `timeout` layered override

## Problem framing

**Who is affected:** Any workflow author who needs per-step or per-backend timeout control. Currently,
all steps use a workflow-level timeout (`Workflow.timeout`, default 2 min) or the global default
(`Defaults.timeout`, default 300s). There is no way to say "Codex steps get 60s, synthesis steps get
10s" without splitting workflows.

**Current behaviour vs wanted behaviour:** The `StepContext.timeout` field exists and `BackendError::Timeout`
is defined, but the resolution logic is duplicated across two paths. In the workflow runner,
`step.timeout.or(workflow.timeout)` is used; in the standalone dispatch path,
`backend_config.timeout.or(config.defaults.timeout)` is used. Neither path implements the full
layered chain **step > backend > global**, and all timeout values are plain integers (seconds or
milliseconds) rather than human-friendly strings like `"30s"` or `"5m"`.

**Why now:** CLO-371 already landed the `StepContext` trait with a `timeout` field. CLO-384 closes
the gap by implementing the correct three-layer resolution at `StepContext` materialisation,
adding string-duration parsing, and making `Defaults.timeout` optional so "no timeout anywhere"
falls back to the backend's built-in default.

---

## Existing code

### Files (per issue)

| File | Role |
|---|---|
| `src/config.rs` | `Defaults.timeout: u64` (seconds, default 300), `BackendConfig.timeout: Option<u64>` (seconds) |
| `src/workflow.rs` | `Workflow.timeout: Option<u64>` (ms), `Step.timeout: Option<u64>` (ms), `step_context()` builder, `Workflow::step_timeout()` |
| `src/backend/context.rs` | `StepContext.timeout: Option<Duration>` — already present |
| `src/backend/mod.rs` | `BackendError::Timeout { message, elapsed_ms }` — already present; `step_context_for_backend()` (backend-dispatch path), `run_query_with_config()` timeout wrapper, `NO_TIMEOUT_SECS` sentinel |
| Cargo.toml | No `humantime` / `duration_str` / `serde_with` dependency |

### Resolution gap

Two separate resolution chains exist and are inconsistent:

1. **Workflow runner** (`src/workflow.rs:2255`):
   ```
   step.timeout.or(workflow.timeout)  // milliseconds, ignores backend config & global defaults
   ```

2. **Standalone dispatch** (`src/backend/mod.rs:329`):
   ```
   backend_config.timeout.or(config.defaults.timeout)  // seconds, ignores step/workflow config
   ```

Neither implements the required **step > backend > global** chain. The `Defaults.timeout` is
`u64` (not `Option<u64>`), so global always has a value (300s), violating the issue's requirement
that it be `Option<Duration>` so the chain can fall through to each backend's own built-in default.

### Duration string support

All timeout fields use raw integers (seconds or milliseconds). The issue specifies string forms
like `"30s"`, `"5m"`. No existing crate handles this.

### Baseline score

**6/10** — The structure (`StepContext.timeout`, `BackendError::Timeout`) is in place, saving
~40% of the work. But the resolution logic is split and inconsistent, string parsing is absent,
and `Defaults.timeout` needs to become optional. The refactoring is mechanical and low-risk.

---

## Approaches

### Approach A: humantime-serde + refactor resolution

Add `humantime-serde` and `serde` (already present) to Cargo.toml. Swap the integer fields to
`Option<humantime_serde::Serde<Duration>>` with `#[serde(with = "humantime_serde")]` or a custom
wrapper. Refactor the two resolution paths into a single `effective_timeout(ctx, config, backend_name)`
function that implements the `step > backend > global` chain, called from both `step_context()`
and `step_context_for_backend()`.

**Pros:**
- `humantime-serde` is mature (used by Tokio ecosystem), handles `"30s"`, `"5m"`, `"1h"` etc.
- Single resolution function eliminates inconsistency.
- `Defaults.timeout` becomes `Option<Duration>`, enabling the fall-through-to-backend-default path.

**Cons:**
- Breaking change for existing `lok.toml` files that use raw integers for `defaults.timeout` and
  `backends.*.timeout`. Mitigation: accept both via a custom deserializer (int or string).
- Small dependency addition.

**Effort:** M
**Risk:** Low

### Approach B: Custom serde deserializer only, keep split paths

Write a custom `deserialize_duration` that accepts both integer (seconds) and string (`"30s"`)
formats. Apply it to each field. Keep the two resolution paths but fix them independently.

**Pros:**
- Zero new dependencies.
- Fine-grained control over the TOML deserialization format.

**Cons:**
- Does not fix the duplicated/inconsistent resolution logic.
- Custom deserializers are boilerplate-heavy for every field.
- Maintaining two paths doubles the test surface.

**Effort:** M
**Risk:** Medium (paths could drift again)

### Approach C: serde_with crate with DurationSeconds + DurationString mix

Use `serde_with`'s `DurationSeconds` and a custom string format to accept both forms.

**Pros:**
- Battle-tested serde utilities.
- Less boilerplate than Approach B.

**Cons:**
- `serde_with` does not directly support `"30s"` / `"5m"` strings out of the box (only ISO 8601
  durations). Would still need a custom wrapper.
- Heavier dependency than `humantime-serde` alone.
- Same split-path problem as Approach B.

**Effort:** L-M
**Risk:** Medium

---

## Chosen approach: **Approach A**

**Reason:** It's the only approach that both (a) adds string-duration parsing and (b) consolidates
the two inconsistent resolution paths into a single function. The backward-compatibility concern
(breaking integer-format `lok.toml` users) is addressed by accepting both integer and string
formats via a custom union deserialiser that wraps `humantime-serde`. No existing production
`lok.toml` files are known to rely on the exact integer format, but the dual-format deserialiser
makes the change safe regardless.

### Plan

1. Add `humantime-serde` to `Cargo.toml`.
2. Write a `duration_maybe_uint` deserialise helper that tries string first (humantime), then
   integer (seconds). Can be placed in `src/utils.rs`.
3. Change `Defaults.timeout` from `u64` to `Option<Duration>` with `#[serde(default, deserialize_with = "...")]`.
4. Change `BackendConfig.timeout` from `Option<u64>` to `Option<Duration>` with the same deserialiser.
5. Change `Step.timeout` from `Option<u64>` (ms) to `Option<Duration>` with the same deserialiser.
6. Note: `Workflow.timeout` currently stores milliseconds. Should it also become `Option<Duration>`?
   The issue says "backend > global" not "workflow > backend". We'll keep `Workflow.timeout` as-is
   (ms) or migrate it to Duration too, with a millisecond integer fallback for backward compat.
7. Write a single `effective_timeout(step, backend_name, config) -> Duration` function in
   `src/backend/mod.rs` implementing the chain. Call it from `step_context()` (workflow.rs) and
   `step_context_for_backend()` (backend/mod.rs).
8. Update tests that hardcode integer timeout values.

### Risk assessment

- **String parsing:** `humantime-serde` is well-tested; the dual-format deserialiser is ~10 lines.
- **Defaults.timeout optional:** The change from `u64` to `Option<Duration>` with `#[serde(default)]`
  means old configs without the field get `None` (safe), and old configs with an integer get parsed
  correctly by the dual-format deserialiser. The default in Rust code becomes `Option<Duration>::None`.
- **Workflow.toml backward compat:** Step-level `timeout` currently stores milliseconds as `u64`.
  Changing to string form breaks existing workflow TOML files. Mitigation: accept both
  `timeout = "30s"` and `timeout = 30000` via the same dual-format helper (the integer branch
  interprets raw numbers as milliseconds for step/workflow contexts, seconds for config contexts).
  Or, more simply: keep step.timeout as milliseconds integer but add an alias/new field for
  string form. Simpler: dual-format deserialiser that interprets integers as milliseconds (matching
  current behaviour) and strings as humantime.
