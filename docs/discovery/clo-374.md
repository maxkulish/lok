# CLO-374 Discovery Report: Per-step sandbox routing for Codex and Gemini backends (FR-21)

**Date:** 2026-05-18
**Source PRD:** `docs/prds/prd-phase-2-predictable-cli-execution-v5.md` §FR-21, §9 release plan step 8

---

## 1. Problem framing

**WHO:** Lok users running multi-step workflows where some steps need read-only access (e.g., `hunt`, `audit`) and others need write access (e.g., `apply_edits`). Currently, Codex's sandbox mode is hardcoded to `read-only` in its default args, and Gemini has no sandbox/approval-mode support at all.

**WHAT:** The `CodexBackend` constructor hardcodes `["exec", "--json", "-s", "read-only"]` as default args. The `GeminiBackend` has no concept of approval mode. Neither backend consults `StepContext.sandbox` at query time. This means every step that uses Codex gets `read-only` regardless of whether it needs to write files.

**WHAT WE WANT:** Any step can declare `sandbox: workspace-write` (for `apply_edits`) or `sandbox: read-only` (for `hunt`/`audit`), and the backend translates that into the correct CLI flag at query time. Default behaviour is unchanged (`read-only`).

**WHY NOW:** The apply-and-verify pipeline (CLO-205/210/211) is already shipped. Without per-step sandbox, `apply_edits` steps cannot use Codex as the backend because Codex's default `-s read-only` blocks file writes. This is the final plumbing step to make per-step configuration flow end-to-end (FR-21 in the v5 release plan, step 8).

## 3. Identified approaches

### Approach A: Thread sandbox through `step_context()` + per-backend arg injection

**Summary:** Add `sandbox: Option<SandboxMode>` to `Step`, thread it via `step_context()` into `StepContext`, then have each backend's `query()` method check `ctx.sandbox` and inject the appropriate CLI flag at runtime.

**Pros:**
- Uses the existing `StepContext` channel already designed for this purpose
- Each backend handles its own arg construction (Codex `-s` vs Gemini `--approval-mode`)
- Backward compatible — `None` preserves current behaviour

**Cons:**
- Codex needs to build args dynamically instead of using stored `self.args` verbatim
- Gemini shell command string needs conditional appending (slightly more fragile)
- Minor: `--ephemeral` is also part of the recommended default args from `codex-quick-ref.md`

**Effort:** S (small)
**Risk:** Low

### Approach B: Add sandbox to each backend's constructor

**Summary:** Instead of checking `ctx.sandbox` at query time, pre-compute the sandbox mode during `new()` and store it. Remove sandbox from `StepContext`.

**Pros:**
- Simpler query-time logic (no branching in `query()`)
- Sandbox is a static config concern, not per-query

**Cons:**
- Sandbox IS per-query (per-step), not per-backend — a single backend instance may handle multiple steps with different sandbox levels
- Violates the StepContext design intent (CLO-371)
- `run_query_with_config` has no Step in scope (non-Step call sites)

**Effort:** M (medium)
**Risk:** Medium

---

## 4. Chosen approach

**Approach A: Thread sandbox through `step_context()` + per-backend arg injection.**

This is the only approach consistent with the StepContext design already shipped in CLO-371. The `sandbox` field exists on `StepContext` but is never populated — this approach connects the final wire. It keeps per-backend sandbox translation logic close to each backend and preserves the per-step granularity that FR-21 requires.

## 6. Implementation outline

### 6.1 `Step` struct — add `sandbox` field

```rust
/// Sandbox mode for subprocess backends (FR-21).
/// None = backend default (read-only).
#[serde(default, deserialize_with = "deserialize_sandbox_mode")]
pub sandbox: Option<SandboxMode>,
```

### 6.2 `step_context()` — thread sandbox

```rust
fn step_context<'a>(...) -> backend::StepContext<'a> {
    backend::StepContext {
        sandbox: step.sandbox,
        timeout: workflow.step_timeout(step).map(Duration::from_millis),
        ..backend::StepContext::from_prompt(prompt, cwd, step.model.as_deref())
    }
}
```

### 6.3 `CodexBackend::query` — use `ctx.sandbox`

Instead of `cmd.args(&self.args)`, build args dynamically:
- Start with `self.args` as base (preserves custom config)
- If no custom args, use `["exec", "--json", "--ephemeral", ...]` recommended defaults
- Append `-s <mode>` based on `ctx.sandbox` (fallback to `read-only`)
- Keep `--model` override logic unchanged

### 6.4 `GeminiBackend::query` — use `ctx.sandbox`

In the shell command string:
- Append `--approval-mode plan` for `ReadOnly`
- Append `--approval-mode auto_edit` for `WorkspaceWrite`
- Append `--approval-mode yolo` for `DangerFullAccess`
- No flag when `None`

### 6.5 Tests

Add argv-capture tests using a mock/test helper:
- `step.sandbox: workspace-write` → Codex argv contains `-s workspace-write`
- Missing `step.sandbox` → Codex argv contains `-s read-only`
- `step.sandbox: read-only` → Gemini argv contains `--approval-mode plan`
- `step.sandbox: workspace-write` → Gemini argv contains `--approval-mode auto_edit`
