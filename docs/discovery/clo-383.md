# Discovery Report: CLO-383 — FR-22: `apply_edits=true` Defaults Codex Sandbox to `workspace-write`

## Problem Framing

The workflow engine's `apply_edits` feature parses JSON file-edits from LLM output and writes them to disk. However, when a workflow step has `apply_edits = true` with no explicit `sandbox`, the backend still runs under its default restricted permissions: Codex defaults to `-s read-only` (FR-21), and Gemini emits no `--approval-mode` flag at all. The step is therefore authorised to demand edits but sandboxed from making them, causing silent failure.

**WHO**: Workflow authors using `apply_edits = true` with Codex or Gemini backends.
**WHAT**: Today `apply_edits = true` + no explicit `sandbox` → Codex gets `-s read-only`, Gemini gets no approval flag. The LLM may emit file edits but the backend cannot write them. Desired: when `apply_edits = true` and no sandbox is explicitly set, the step should default to `workspace-write` (Codex) / `auto_edit` (Gemini). If a `sandbox` IS explicitly set, it always wins.
**WHY NOW**: FR-21 (CLO-374) already wired per-step `StepContext.sandbox` through both Codex and Gemini backends. The plumbing exists; the only missing piece is the `apply_edits → workspace-write` defaulting rule. This is the final sandbox-level gap in the predictable CLI execution pipeline.

## Existing Code

### `src/workflow.rs`

- **`Step` struct** (line ~249): carries both `apply_edits: bool` and `sandbox: Option<SandboxMode>`. `apply_edits` is toggled in the main execution loop (~line 2302) but is never used upstream when building `StepContext`.
- **`step_context()`** (lines 160–179): builds `StepContext` from `Step`. Sets `sandbox: step.sandbox` directly — no `apply_edits` awareness:
  ```rust
  fn step_context<'a>(step: &'a Step, workflow: &Workflow, prompt: &'a str, cwd: &'a Path) -> backend::StepContext<'a> {
      backend::StepContext {
          sandbox: step.sandbox,   // ← passes through raw step value
          timeout: workflow.step_timeout(step).map(std::time::Duration::from_millis),
          ..backend::StepContext::from_prompt(prompt, cwd, step.model.as_deref())
      }
  }
  ```
- `StepContext::from_prompt()` (in `context.rs`) initializes all extended fields (`sandbox`, `schema`, `options`, `timeout`) to `None`.

### `src/backend/context.rs`

- **`StepContext`** (lines 12–37): a `Copy` carrying struct with `sandbox: Option<SandboxMode>` but **no `apply_edits`** field. Adding a `bool` is trivial because the struct is Copy and all fields are POD.
- **`SandboxMode`** (lines 61–66): enum with three variants (`ReadOnly`, `WorkspaceWrite`, `DangerFullAccess`). Already wired through serde as kebab-case.

### `src/backend/codex.rs`

- **`build_argv_prefix()`** (lines 40–67): receives `sandbox: Option<SandboxMode>` and **defaults to `ReadOnly`** when `None`:
  ```rust
  let mode = sandbox.unwrap_or(super::SandboxMode::ReadOnly);
  ```
  Then emits `-s read-only` / `-s workspace-write` / `-s danger-full-access`.
- **`query()`** (line ~95): passes `ctx.sandbox` unmodified to `build_argv_prefix`. No `apply_edits` awareness.
- **Tests** (lines 178–242): Cover sandbox mapping for explicit values and `None → read-only`, but no test exercises the `apply_edits` defaulting matrix.

### `src/backend/gemini.rs`

- **`build_shell_cmd()`** (lines 130–154): maps `sandbox` to `--approval-mode` flag. When `None`, emits **no approval flag at all**:
  ```rust
  None => "",
  ```
- **`query()`** (line ~180): passes `ctx.sandbox` to `build_shell_cmd`. No `apply_edits` logic.
- **Tests** (lines 305–362): Cover sandbox mapping but not the new `apply_edits` defaulting.

### `src/config.rs`

- Default Codex args test (line ~420): confirms no `-s` in default config-level args, proving per-step injection is already the intended pattern.

## Baseline Score: 9/10

No rewrite needed. FR-21 (CLO-374) landed the per-step sandbox infrastructure; we only need to layer the `apply_edits` resolution on top. The change is additive, touch-points are limited (`StepContext` + two backend query paths + `step_context()` builder), and the test surface is well-isolated.

## Approaches Identified

### Approach A: Add `apply_edits` to `StepContext`, resolve in backends

Add `apply_edits: bool` to `StepContext`. In `step_context()`, set it from `step.apply_edits`. In `CodexBackend::build_argv_prefix()` and `GeminiBackend::build_shell_cmd()`, compute an **effective sandbox**:
- If `apply_edits && sandbox.is_none()` → `Some(WorkspaceWrite)`
- If `apply_edits && sandbox == Some(ReadOnly)` → log warning and pass through as-is
- Otherwise → pass through unchanged

- **Pros**:
  - Backend-aware logic lives in backends — proper separation of concerns.
  - Centralises both defaulting and the warning for `apply_edits + read-only` in the code that understands what `read-only` means.
  - Matches the existing pattern where `StepContext` carries per-step concerns (`model`, `timeout`, `schema`, `options`).
  - If a future backend (e.g. a new subprocess backend) needs similar defaulting, the field is already available.
- **Cons**:
  - Changes `StepContext` struct (one `bool` — struct is `Copy`, so zero runtime cost).
  - Two backends implement the same defaulting rule; a shared helper could be added if a third backend needs it.
- **Effort**: S
- **Risk**: Low

### Approach B: Resolve in `workflow.rs::step_context()`

In `step_context()`, set `sandbox: Some(SandboxMode::WorkspaceWrite)` when `step.apply_edits == true && step.sandbox.is_none()`. Log a warning when `apply_edits == true && step.sandbox == Some(ReadOnly)`. No `StepContext` changes needed.

- **Pros**:
  - Single point of defaulting logic in one file.
  - No StepContext struct changes; no backend code touched.
- **Cons**:
  - Backend-agnostic defaulting: `sandbox` is set for **all** backends even though only Codex/Gemini consume it. While harmless today (Ollama/Claude/Bedrock ignore `sandbox`), it's semantically imprecate and risks surprising a future backend that adds sandbox support.
  - The `apply_edits + read-only` warning belongs in backend code (workflow.rs shouldn't know that `read-only` means "can't write files" for a CLI backend).
  - A step can have multiple backends in consensus mode; the `step_context()` builder is called once but the defaulting intent is backend-specific.
- **Effort**: S
- **Risk**: Low

### Approach C: Add a `Backend` trait method for sandbox defaulting

Introduce `fn default_sandbox(&self, apply_edits: bool) -> Option<SandboxMode>` on the `Backend` trait with a default implementation returning `None`. Codex and Gemini override it; Ollama/Claude/Bedrock keep the default. `step_context()` consults the backend(s).

- **Pros**:
  - Each backend declares its own defaulting policy — most extensible.
  - No StepContext changes required.
- **Cons**:
  - Consensus steps have multiple backends; calling the trait method for each backend is confusing because `step_context()` produces **one** `StepContext`.
  - Over-engineered for a rule that only two backends need and that behaves identically (`None + apply_edits → WorkspaceWrite`).
  - Trait churn (new method, default impl) for a trivial rule.
- **Effort**: M
- **Risk**: Low

## Chosen Approach: A

**Reason**: The `Backend` trait already receives `StepContext` as the single source of per-step truth. Adding `apply_edits: bool` to that context is the most natural extension of an existing pattern. Backend-specific defaulting then lives where it belongs: in `CodexBackend` and `GeminiBackend`. The `step_context()` builder remains transparent and backend-agnostic. This cleanly separates materialisation (workflow.rs) from interpretation (backends), consistent with how `model`, `timeout`, and `schema` already flow through the same pipe.
