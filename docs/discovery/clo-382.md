## Problem Framing

The Gemini backend (`src/backend/gemini.rs`) today only parses stdout by skipping leading lines via `skip_lines`. When Gemini CLI is invoked with `--output-format json`, it emits a single JSON envelope containing the response text **and** token-usage statistics (`stats.promptTokenCount`, `stats.candidatesTokenCount`, and optionally `stats.cachedContentTokenCount`). However, `GeminiBackend::query()` returns `usage: None` on every call, leaving `StepResult.usage` unpopulated even though the data is present in the raw envelope. This breaks FR-25a's end-to-end token observability for the Gemini path.

**WHO**: Workflow authors running workflows with `backends.gemini` as the primary or consensus backend.
**WHAT**: Gemini queries report no token usage even when the CLI returns structured stats. The `parse_output()` method blindly strips the first line and discards the JSON envelope, throwing away the `stats` block.
**WHY**: This is one of the last two backends (Gemini + Claude CLI) that do not yet populate `QueryOutput.usage`. CLO-370 added `StepResult.usage` and CLO-378 added `TokenUsage::cached_tokens`/`reasoning_tokens`, so Gemini is the only remaining major CLI backend without usage extraction.

## Approaches Identified

### Approach A: Parse JSON envelope after child exit, fallback to text
After `output.status.success()`, attempt `serde_json::from_str::<serde_json::Value>(&stdout)`. On success, read `.stats.promptTokenCount` / `.stats.candidatesTokenCount` / `.cachedContentTokenCount` and `.response`. On any parse/field-missing failure, fall back to the existing `parse_output()` text behavior (skip lines + return raw text) with `usage = None`.

- **Pros**:
  - Minimal change to existing flow.
  - Zero risk of regressing text extraction â€” malformed JSON falls back to current behavior.
  - Matches Codex's existing pattern (parse JSONL, then `.with_usage(parsed.usage)`).
- **Cons**:
  - Pays serde_json parse cost even when no stats present.
  - Two extraction paths increase surface area mildly.
- **Effort**: S
- **Risk**: Low

### Approach B: Always parse JSON envelope, treat non-JSON as error
Assume all Gemini CLI invocations always return `--output-format json`. Replace `parse_output()` entirely with a strict serde parse; treat non-JSON stdout as `BackendError::Parse`.

- **Pros**:
  - Single code path.
  - Cleaner contract.
- **Cons**:
  - Requires guaranteeing JSON mode on every invocation (config-level flag).
  - Breaks backward compat for users who rely on text mode.
  - Higher risk: any non-JSON output (warnings, version banners) becomes fatal.
- **Effort**: S
- **Risk**: Medium

### Approach C: Parse during command parsing, not after
Change `build_shell_cmd` to optionally append `--output-format json`, then return a structured result with a `GeminiOutput` enum from the query method.

- **Pros**:
  - Type-safe at compile time.
- **Cons**:
  - Overkill for a single-field extraction.
  - Requires changing return type or adding an enum where `String` suffices.
- **Effort**: M
- **Risk**: Low

## Chosen Approach: A (Parse JSON envelope with text fallback)

**Reason**: The issue explicitly states "If JSON envelope is malformed: step still surfaces the textual result via existing extraction logic; usage = None." This matches Approach A exactly. It also aligns with Codex's existing pattern of parsing structured output after the child exits and wiring `.with_usage(...)`. No trait changes, no config changes, no new dependencies.
