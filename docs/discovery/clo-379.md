# Discovery Report: CLO-379 — FR-3a Event-driven Codex JSONL parser

## Problem framing

The `CodexBackend` in `src/backend/codex.rs` extracts the final agent response from Codex `exec --json` JSONL streams using a brittle substring match (`line.contains("\"type\":\"item.completed\""`) && `line.contains("agent_message")`). This heuristic:

- Returns the **first** `agent_message` found anywhere in the stream, ignoring turn boundaries — a multi-turn or intermediate-reasoning stream would return the wrong item.
- Fails silently on `turn.failed` or top-level `error` events, falling back to raw output instead of signaling a backend error.
- Cannot distinguish missing `agent_message` (older Codex / interrupted) from success, again returning raw output.
- Is fragile to JSON whitespace or field-order changes and discards `usage` tokens completely.

The trigger is that FR-40 (CLO-373) has landed real JSONL fixtures on `main`, making it possible to replace the heuristic with a robust event-driven parser that understands Codex stream semantics.

## Existing code

**File: `src/backend/codex.rs`**

Current `parse_output` method (~L95):

```rust
fn parse_output(&self, output: &str) -> String {
    for line in output.lines() {
        if line.contains("\"type\":\"item.completed\"") && line.contains("agent_message") {
            if let Ok(json) = serde_json::from_str::<serde_json::Value>(line) {
                if let Some(text) = json.get("item").and_then(|i| i.get("text")).and_then(|t| t.as_str()) {
                    return text.to_string();
                }
            }
        }
    }
    output.to_string()
}
```

**Observations:**
- **Baseline score: 2/10.** Works for a single-turn happy path with exact JSON formatting, but fails on every other dimension the PRD requires (multi-turn correctness, error propagation, parse failure detection, forward compatibility, token usage extraction).
- The `query()` method returns `BackendError` variants from `src/backend/mod.rs` including `Parse { message: String }` and `ExecutionFailed { message: String, exit_code: Option<i32> }`, but `parse_output` currently uses `String` return type — it cannot signal errors. Changing the return type to `Result<String, BackendError>` is a required internal refactor.
- Fixtures exist at `tests/fixtures/codex/` (CLO-373 / FR-40): `turn-completed.jsonl`, `turn-failed.jsonl`, `multi-turn-reasoning.jsonl`, `missing-agent-message.jsonl`.
- Existing tests in `tests/codex_fixtures.rs` validate fixture shape (valid JSONL, known event types, terminal events, sensitive-data scrub) but do **not** test the actual `CodexBackend::parse_output` method. New unit tests in `src/backend/codex.rs` will be needed.
- `BackendError::Backend` does not exist in the current enum; the PRD/ticket says "short-circuits to `BackendError::Backend` with the event's `error.message`". The closest variant is `ExecutionFailed { message, exit_code }`. A follow-up question: should we add `Backend` variant or reuse `ExecutionFailed`? **Decision:** Use `ExecutionFailed { message, exit_code: None }` to keep the change additive and avoid enum modification in this PR. This matches the existing `BackendError` surface. (Alternatively, if the project prefers adding `Backend`, that should be noted as a minor variant addition in `mod.rs`.)

## PRD reference

`docs/prds/prd-phase-2-predictable-cli-execution-v5.md` §4 FR-3a defines:

- Event model: `thread.started`, `turn.started`, `turn.completed`, `turn.failed`, `item.started`, `item.completed`, `error`
- Extraction rule: `agent_message` item from the **last** `turn.completed`
- `turn.failed` → `BackendError::Backend` (we'll use `ExecutionFailed`)
- Unknown event types → logged at debug, skipped
- Unparseable lines → logged at debug; if no `turn.completed` by stream end → `BackendError::Parse`
- FR-3b (`-o` / `--output-last-message`) is **out of scope**

## Approaches identified

### Approach 1: Event-driven serde enum (chosen)

Define `CodexEvent` as a `#[serde(tag = "type")]` enum over all documented event types, with `#[serde(other)]` for forward compatibility. Accumulate `agent_message` items from the current turn; on `turn.completed`, return the last accumulated text and usage. On errors, return `BackendError::ExecutionFailed`.

**Pros:**
- Type-safe, compile-time enforcement of event field access.
- Forward-compatible via `Unknown` variant.
- Exactly matches the PRD event model and acceptance criteria.
- Can extract usage tokens (`usage` from `turn.completed`) for future FR-25 work.

**Cons:**
- ~60–80 new lines of event type definitions.
- Requires refactoring `parse_output` to return `Result`.

**Effort:** S
**Risk:** Low

### Approach 2: Streaming `serde_json::Value` matcher

Parse each line as `serde_json::Value` and manually match on `"type"` string without defining a typed enum. Accumulate state in local variables.

**Pros:**
- No new structs/enums needed.
- Slightly less boilerplate.

**Cons:**
- Still manual and error-prone JSON navigation.
- No compile-time safety on field names or types.
- Harder to extend for future event types.

**Effort:** S
**Risk:** Medium

### Approach 3: Regex-based extraction

Replace `contains()` with regex patterns for each event type.

**Pros:**
- Minimal code change.

**Cons:**
- Still brittle; regex on JSON is a well-known anti-pattern.
- Does not solve the fundamental semantic problem of understanding event boundaries.
- Forward compatibility is harder than with `#[serde(other)]`.

**Effort:** XS
**Risk:** High

## Approach chosen

**Approach 1 — Event-driven serde enum.**

Reason: The PRD already specifies the exact event model; serde provides compile-time safety and forward compatibility. The fixture corpus gives us a concrete test target. The additional type definitions are a net win for maintainability, especially since FR-25 (token usage) and future Codex releases will touch this parser. Approach 2 is inferior on type safety and Approach 3 is semantically wrong.
