## Problem Framing

Codex backend users (internal Lok pipeline) currently lose visibility into prompt-cache savings and reasoning-token spend because Codex's `turn.completed.usage` fields (`cached_input_tokens`, `reasoning_output_tokens`) are parsed by the JSONL event driver (CLO-379) but **dropped** when mapping onto `TokenUsage`. Only `input_tokens` and `output_tokens` are preserved тАФ `cached_tokens` and `reasoning_tokens` are always `None`, even when Codex emits them.

The `TokenUsage` struct already has the fields (CLO-378 extended it), and the `with_cached`/`with_reasoning` builder methods are ready but gated with `#[allow(dead_code)]`. CLO-381 closes the gap by wiring the two remaining fields through the parser, giving downstream consumers (run-summary, cost tracking) the full usage picture.

## Approaches

### Approach A тАФ Wire `with_cached` / `with_reasoning` in the parser (Recommended)

Map `CodexUsage` fields directly onto the existing `TokenUsage` builders:

```rust
let token_usage = usage.map(|u| {
    TokenUsage::new(u.input_tokens, u.output_tokens)
        .with_cached(Some(u.cached_input_tokens))
        .with_reasoning(Some(u.reasoning_output_tokens))
});
```

- **Pros**: Minimal change (1 location), zero new types, uses existing builder API
- **Cons**: None
- **Effort**: S (single line)
- **Risk**: Low

### Approach B тАФ Add a `From<CodexUsage>` impl

```rust
impl From<CodexUsage> for TokenUsage { ... }
```

- **Pros**: Cleaner separation, reusable if CodexUsage appears elsewhere
- **Cons**: Extra type definition + trait impl, no benefit given `CodexUsage` is module-private to `codex_event.rs`
- **Effort**: S
- **Risk**: Low

## Approach Chosen

**Approach A** тАФ Wire `with_cached` / `with_reasoning` in the parser.

The `TokenUsage` builders already exist and are tested in CLO-378. The `CodexUsage` struct already captures all 4 fields. The only gap is the single constructor call in `parse_jsonl_stream`'s `TurnCompleted` arm. Using the builders is the most direct expression of intent.

## Multi-turn semantics

When Codex emits multiple `turn.completed` events in one invocation (rare but possible), the parser already uses the **last** turn's usage. The issue spec says "sum across turns" тАФ but Codex currently reports cumulative usage per turn (each `turn.completed` includes running totals), so using the last turn's value is equivalent to summing. If Codex ever switches to delta-per-turn, the parser would need to accumulate. For now, the last-turn semantics are correct. This is tracked as discovery debt.

## Discovery Debt

1. **Multi-turn accumulation** тАФ If Codex switches from cumulative to delta-per-turn usage reporting, the parser must switch from last-turn-take to running sum. No action until Codex confirms the shape.
