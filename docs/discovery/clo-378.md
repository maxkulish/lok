# Discovery Report: CLO-378 — FR-25b TokenUsage Extension

## Problem Framing

**WHO**: Downstream consumers of `StepResult.usage` — run-summary aggregation (FR-28), JSON output mode (FR-29), rs-wisper crate that depends on `TokenUsage` shape. Also FR-25 (Codex usage extraction, CLO-381) and FR-26 (Gemini backend, CLO-382) which need the extended fields to exist before they can wire upstream data.

**WHAT today**: `TokenUsage` has 3 `u32` fields (`prompt_tokens`, `completion_tokens`, `total_tokens`). Anthropic's `cache_read_input_tokens` and Codex's `cached_input_tokens` / `reasoning_output_tokens` exist in API responses but are flattened to `total_tokens` because the struct has no dedicated fields. FR-25a (CLO-370) wired `StepResult.usage` but with this lean shape.

**WHAT desired**: Add `cached_tokens: Option<u32>` and `reasoning_tokens: Option<u32>`. Backends that don't report them leave them `None`. `total_tokens` stays `prompt + completion` unchanged.

**WHY now**: Smallest possible follow-up to CLO-370. Precondition for FR-25 (CLO-381) and FR-26 (CLO-382). Additive; zero caller breakage. Release plan step 4 in PRD §9.

## Approaches

### Approach 1: Minimal struct extension (RECOMMENDED)

Add two `Option<u32>` fields to `TokenUsage`. Keep `new()` defaulting them to `None`. Fix the one struct-literal test.

**Pros**:
- ~10 lines of code changes
- Zero breakage to 12 of 13 construction sites
- Matches PRD §4 FR-25b exactly
- Enables FR-25 / FR-26 immediately

**Cons**:
- serde derives not added (deferred to FR-29 if needed)

**Effort**: S (one afternoon)
**Risk**: Low

### Approach 2: Struct extension + serde derives

Same as Approach 1 but also adds `#[derive(Serialize, Deserialize)]` with `#[serde(default)]` on the new optionals.

**Pros**:
- One fewer follow-up ticket for FR-29
- JSON output mode can serialize TokenUsage directly

**Cons**:
- `StepResult` doesn't derive serde today — adding it only to `TokenUsage` changes the crate's public API surface
- Unnecessary until FR-29 lands
- Requires adding `serde` to the dependency (if not already re-exported)

**Effort**: S
**Risk**: Low (but premature)

### Approach 3: Builder + extension

Replace the constructor with a builder to handle growing optional fields.

**Pros**:
- Scales to future fields (e.g. FR-30 token budget)

**Cons**:
- Breaks all existing call sites
- Over-engineered for 2 optional fields

**Effort**: M
**Risk**: Medium
