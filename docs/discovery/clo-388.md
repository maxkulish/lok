# Discovery Report: CLO-388 — FR-9a + FR-10 + FR-15: Engine warmup + HealthCache + sync is_available cache-only

## Problem Framing

**WHO**: Users running workflows, direct queries, or system diagnostics (`lok run`, `lok ask`, `lok doctor`), and downstream steps executing on selected backends.

**WHAT today**: Currently, `Backend::is_available()` is a synchronous, un-cached function that is called dynamically whenever backends are retrieved via `get_backends()`. For most backends (e.g. Gemini, Codex, Claude), this triggers a filesystem/process syscall using `which::which` to check PATH. For Ollama, it unconditionally returns `true` (which is a known lie). For Bedrock, it checks environment variables and verifies the existence of `~/.aws/credentials`. There is no background warmup pipeline, and no shared caching. Consequently:
1. System/process syscalls (such as `which`) are invoked repeatedly on hot execution paths.
2. We cannot verify real server connectivity (Ollama) or actual authentication dynamically without blocking the synchronous runtime.
3. Every step execution risk-checks backend availability dynamically.

**WHAT desired**: 
1. `Engine::warmup_backends()` runs every enabled backend's `health_check()` in parallel (fanning out via `futures::future::join_all` / `tokio::join_all`) before execution or diagnostics begin.
2. A shared thread-safe memory cache `HealthCache` stores `HealthStatus` results per backend for the duration of a `lok` invocation.
3. Every backend's sync `is_available()` becomes a **cache-only** memory lookup. It returns `false` until warmup populates it, and **never** makes network or process calls (no syscalls).
4. `lok run`, `lok ask`, and `lok doctor` all call `warmup_backends()` at their startup/entry points.

**WHY now**: This is the foundational health-check infrastructure under PRD §4 "Real Health Checks & Version Detection". It decouples live probe checks (which are asynchronous/network/process calls) from the synchronous `is_available()` interface. Subsequent per-backend probes (FR-11/12/13/13a) will build on this cache instead of having to deal with synchronous execution path constraints.

## Existing Code

### `Backend::is_available` implementations today

Each backend currently implements its own synchronous check:
* **Ollama (`src/backend/ollama.rs:163`)**: Always returns `true`.
* **Gemini (`src/backend/gemini.rs:257`)**: Runs `which::which(&self.command).is_ok()`.
* **Codex (`src/backend/codex.rs:232`)**: Runs `which::which(&self.command).is_ok()`.
* **Claude (`src/backend/claude.rs:301`)**: Mode-dependent (API checks env var, CLI checks `which::which`).
* **Bedrock (`src/backend/bedrock.rs:203`)**: Checks environment variables or AWS credentials file existence.

### `Backend::health_check` default implementation (`src/backend/mod.rs:288`)

```rust
async fn health_check(&self) -> std::result::Result<HealthStatus, BackendError> {
    if self.is_available() {
        Ok(HealthStatus)
    } else {
        Err(BackendError::Unavailable {
            message: format!("Backend {} is not available", self.name()),
        })
    }
}
```

* **Circular Dependency Warning**: If `is_available()` is changed to read only from the cache, the default `health_check` implementation (which calls `self.is_available()`) will fail on an empty cache!
* **Resolution**: The live `health_check` probes must perform the *actual* active system/network checks previously done by `is_available()`, leaving `is_available()` to be a pure cache lookup.

## Approaches

### Approach 1: Global static OnceLock-based HealthCache (RECOMMENDED)

Define a thread-safe static `OnceLock<RwLock<HashMap<String, HealthStatus>>>` inside the `backend` or `engine` module.

* **Pros**:
  * Does not require changing the `Backend` trait definition or modifying the constructor signatures for all 5 backends.
  * Synchronous `is_available(&self)` can query the static cache instantly without requiring async or borrowing references.
  * Simple and robust.
  * Extremely easy to clear/pre-populate during testing.

* **Cons**:
  * Global state (though completely acceptable and idiomatic for a command-line tool's single-execution lifetime).

### Approach 2: Threading a HealthCache reference to every backend constructor

Add `cache: Arc<RwLock<HashMap<String, HealthStatus>>>` to every backend constructor and struct.

* **Pros**:
  * No global static state.

* **Cons**:
  * Massive signature break. Requires updating constructors and structs across all 5 backends, plus updating `create_backend` and `get_backends`.
  * Increases boilerplate and coupling.

## Discovery Debt

None. The codebase is clean and perfectly set up for this refactoring.

## Verdict

**Ready to Build**. We will proceed with **Approach 1** by:
1. Moving real probes from `is_available()` to `health_check()` on each backend.
2. Implementing `Engine::warmup_backends()` to probe all enabled backends in parallel and cache the results.
3. Modifying `is_available()` to read exclusively from the cache (and return `false` if not found/warmed up).
4. Wiring `Engine::warmup_backends()` into the entrypoints of `lok run`, `lok ask`, and `lok doctor`.
