# Discovery Report: CLO-406 — FR-15a `LOK_HEALTH_TTL` env override for HealthCache TTL

## 1) Problem framing

**WHO:** Backend availability checks and diagnostics in `lok` (especially `lok doctor`, `lok ask`, and `lok run`) and any host that embeds lok in a long-lived process.

**WHAT (today):** `BACKEND_CACHE` in `src/backend/mod.rs` caches `HealthStatus` for configured backends for the entire process lifetime. `is_backend_available()` only checks `HealthStatus.available` from this cache, so long-lived embedders can keep reporting stale availability indefinitely.

**WHAT (desired):** Add `LOK_HEALTH_TTL` as a startup-only override so stale health entries expire after a configured duration, while preserving current behavior for one-shot runs when unset.

**WHY now:** `FR-15a` is unblocked by completed probe work in `CLO-388/389/391/392/395`, and stale cache data is now a practical correctness issue for long-running hosts.

**Constraints:**
- Resolver should be env-only (`LOK_HEALTH_TTL`), no CLI flag.
- Parser should prefer humantime strings (consistent with `CLO-384` timeout conventions).
- `Backend::is_available()` should remain a cache read path.

## 2) Existing code

### `src/backend/mod.rs`
- `CachedBackend` currently stores `backend` + `health: Option<HealthStatus>` at lines around 412–417 (`get_backend_cache` / `BACKEND_CACHE`).
- `create_backend()` inserts entries with `health: None` before probing.
- `Engine::warmup_backends()` probes entries only when `health` is `None` and then writes back `Some(HealthStatus)` (lines around 468 onward).
- `Engine::is_backend_available()` currently reads cached status directly and returns `false` only when absent/unavailable.
- There is no TTL concept for cache entries in this module yet.

### `src/backend/context.rs`
- `HealthStatus` includes `available, version, mode, diagnostic, auth_method, capabilities, unusable_flags, models`.
- Struct has no probe timestamp metadata.

### `src/main.rs`
- Startup warmup already called in `Commands::Ask`, `Commands::Doctor`, and workflow execution (`run_workflow`).
- `Commands::Doctor` renders cached entries and exits with non-zero when any cached backend is unavailable.

### `src/config.rs`
- Existing duration parser helpers already support humantime parsing via `humantime::parse_duration` for config durations.
- No env-based TTL parser exists for cache policy.

### Observability/docs
- Setup guide has a `lok doctor` section with no mention of HealthCache TTL behavior.

## 3) Baseline score

**8/10**

The health cache, warmup integration, and cache-read availability pattern are already in place from earlier tickets. Remaining work is largely localized to cache metadata + env parsing + startup reporting + tests.

## 4) Approaches identified

### Approach A (Recommended): Enrich `CachedBackend` with probe timestamp

**How:**
- Add `checked_at: std::time::Instant` (or `SystemTime`) to `CachedBackend`.
- Parse `LOK_HEALTH_TTL` once at startup in `backend` module.
- On `is_backend_available()`, treat entries older than TTL as unavailable and (optionally) eligible for refresh by next warmup.

**Pros:**
- Minimal surface area.
- Keeps source of truth in cache-owning module.
- Fits existing `BACKEND_CACHE` architecture and call paths.

**Cons:**
- Changes many assertions expecting `CachedBackend` shape in tests.
- Requires clear policy for stale entries (remove vs mark unavailable).

**Effort:** S–M

**Risk:** Medium

### Approach B: Introduce a dedicated `HealthCacheEntry` wrapper

**How:**
- Keep `CachedBackend` for backend instances.
- Add `HealthCacheEntry { status: Option<HealthStatus>, checked_at: Option<Instant> }` and store that as an additional map entry alongside `BACKEND_CACHE`.

**Pros:**
- Avoids structural change in `CachedBackend`.
- Cleaner separation between backend object and probe metadata.

**Cons:**
- Two maps increase coupling risk.
- Higher chance of map drift/staleness between cache and backend maps.

**Effort:** M

**Risk:** Medium

### Approach C: Store TTL on `HealthStatus`

**How:**
- Add `checked_at` timestamp field to `HealthStatus` itself.

**Pros:**
- No new cache entry structs or parallel maps.
- No `CachedBackend` API changes.

**Cons:**
- Mixes transport/cache metadata with semantic health payload.
- Requires touching all test and call sites that compare/construct synthetic statuses.

**Effort:** M

**Risk:** High

## 5) Chosen approach

**Chosen: Approach A (enrich `CachedBackend`).**

**Why:** Keeps TTL enforcement where cache behavior is already implemented, minimizes API noise, and aligns with the ticket’s direction (“source of truth in same module as HealthCache”). A dedicated wrapper map (Approach B) adds unnecessary synchronization risk, and tagging `HealthStatus` (Approach C) dilutes domain meaning and forces broader data-shape churn.

### Chosen implementation sketch (high-level)
- Add a startup helper in `src/backend/mod.rs` that resolves TTL from `LOK_HEALTH_TTL` once, defaulting to a compiled constant.
- Add `checked_at: Option<Instant>` to `CachedBackend` and set it whenever warmup writes `Some(HealthStatus)`.
- `Engine::is_backend_available()` checks freshness before returning `available`.
- `Engine::warmup_backends()` triggers fresh probes for entries that are missing health or stale.
- Add unit tests for parser + expiry behavior.
- Add one `lok setup` note under `lok doctor` with practical guidance on TTL tuning.
