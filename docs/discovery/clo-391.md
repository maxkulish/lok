# CLO-391 Discovery — FR-13a: Claude dual-mode health probe (Api vs Cli)

**Date**: 2026-05-24
**Phase**: discovery

---

## Problem framing

The `lok doctor` command (FR-14, CLO-393) needs a health probe for Claude that works in two modes — when the user configures an API key (Claude API client) and when they use the `claude` CLI binary. Currently the Claude backend's `health_check()` is a thin shell that checks `api_key` non-empty (API mode) or `which::which("claude")` (CLI mode) and returns a bare `HealthStatus::new_available()` — no version info, no flag-support detection, no `mode` discriminator. The doctor renderer would show only one "Claude" row without knowing which mode(s) are configured. The trigger is PRD v5 §4 FR-13a, following the Ollama probe (CLO-389) pattern but adding mode-awareness and offline-cheap API probing.

### Current Claude `health_check()` (lines 308-321)

```rust
async fn health_check(&self) -> Result<HealthStatus, BackendError> {
    let available = match &self.mode {
        ClaudeMode::Api { api_key, .. } => !api_key.expose_secret().is_empty(),
        ClaudeMode::Cli { command, .. } => which::which(command).is_ok(),
    };
    if available {
        Ok(HealthStatus::new_available())
    } else {
        Err(BackendError::Unavailable { .. })
    }
}
```

**Gaps vs FR-13a:**
- API probe does not validate `model` is non-empty
- CLI probe does not run `claude --version` nor parse semver
- CLI probe does not check `claude --help` for `--output-format json` support
- Neither sets `mode: Some("api")` / `Some("cli")` — field doesn't exist yet
- `HealthCache` key is `"claude"` — cannot store two independent results

### Patterns from other backends

| Backend | health_check pattern | sets version? | sets mode? |
|---------|---------------------|---------------|------------|
| Ollama (CLO-389) | HTTP GET /api/version + /api/tags (2s timeout) | Yes | No |
| Codex | `which::which("codex")` | No | No |
| Gemini | `which::which("gemini")` | No | No |
| Bedrock | env var / file existence checks | No | No |
| **Claude (current)** | `which::which` or key non-empty | **No** | **No** |

Claude dual-mode will be the first backend with a `mode` discriminator and the first CLI probe to shell out for version/flag info.

### Baseline score: **6/10**

The `ClaudeMode` enum (Api/Cli) already exists with proper field separation, so no structural refactor is needed. The backend trait, health cache, and test infrastructure are in place from CLO-388/CLO-389. The health_check method itself needs a complete rewrite with:
- New `HealthStatus.mode` field (+ serialization roundtrip)
- Two private probe implementations
- Cache key that includes mode
- 6 new unit tests + test helpers

## Approaches

### Approach A: Inline per-variant probes *(recommended)*

Split `health_check()` into two private methods `probe_api()` and `probe_cli()`, dispatch on `&self.mode` in the single override. Add `mode: Option<String>` to `HealthStatus`. For the cache key, prefix with mode (`"claude::api"`, `"claude::cli"`).

**Pros:**
- One override, one file changed (plus HealthStatus field)
- Follows the existing pattern (Codex/Gemini/Bedrock all use a single override)
- Cache key change is minimal — the warmup loop already uses `backend.name()`
- All probe logic lives inside `ClaudeBackend` where mode fields are directly accessible

**Cons:**
- Slightly harder to unit-test the CLI probe (requires scoping temp bins)
- Version-to-flag-support caching for `--help` output needs care (per-version memoization)

**Effort:** S (2-3 files, ~80 lines)
**Risk:** Low

### Approach B: Extract probe structs

Create `ClaudeApiProbe` and `ClaudeCliProbe` types implementing a common `ClaudeHealthProbe` trait. The backend creates the appropriate probe at construction time and delegates `health_check`.

**Pros:**
- Probe logic is independently testable without backend construction
- Cleaner separation if FR-16 (per-mode capability sets) adds more per-variant methods

**Cons:**
- Premature abstraction for a single method
- Adds a trait + 2 structs + impl blocks for what's currently 10 lines
- The backend already owns the mode data; extracting adds indirection without benefit
- CLO-393 (FR-14 doctor renderer) may need the mode field but not the probe separation

**Effort:** M (3-4 files, ~150 lines)
**Risk:** Low (but more surface area to review)

---

## Chosen approach

**Approach A: Inline per-variant probes.**

The ClaudeMode enum already cleanly separates Api vs Cli state. Adding two private methods inside `ClaudeBackend` is the most natural extension of the existing code — the same pattern every other backend uses. Extracting probe structs (B) would be premature: there's only one consumer (`health_check`), and FR-16 capabilities are a separate concern. Simpler code, smaller diff, faster review.
