# Discovery: CLO-372

## Problem framing
- **Who is affected:** Non-workflow call paths that call `Backend::query` directly (`src/conductor.rs`, `src/spawn.rs`, `src/team.rs`, `src/debate.rs`, and `src/backend/mod.rs::run_query_with_config`).
- **Current behavior:** These call sites construct `StepContext` with `StepContext::from_prompt(..., None)` or equivalent, so explicit model/timeout/profile metadata from command/tool context is frequently dropped.
- **Desired behavior:** Ensure non-`Step` call sites pass a rich `StepContext` with explicit model when available and existing config values for timeout/history/schema/options/sandbox where defined.
- **Why now:** FR-20a already standardized workflow-path context propagation; FR-20b is the remaining non-workflow gap blocking complete parity and consistent observability.
- **PRD source:** Existing phase-2 PRD (`docs/prds/prd-phase-2-predictable-cli-execution-v5.md`) already contains FR-20b coverage; no new per-issue PRD is required.

## Current discovery evidence
- `src/workflow.rs` already builds step-aware contexts with model and timeout (`step_context`, `query_with_model`, and multiple struct-update call sites), so the main gap is bounded to non-step code paths.
- `src/backend/context.rs` confirms `StepContext` supports optional `model`, `timeout`, and passthrough fields already.

### Call sites still passing bare context
1. `src/conductor.rs` tool `query_backend` handler: `StepContext::from_prompt(prompt, cwd, None)`
   - Has backend+prompt+cwd only.
2. `src/spawn.rs`:
   - `plan_with_conductor`: model-less prompt context
   - `plan_with_backend`: model-less prompt context
   - `run`: agent execution loop passes `from_prompt(..., None)`
   - Has task/backend names and global config but no step/model field per task.
3. `src/team.rs`:
   - team suggestion loops call `from_prompt(..., None)`
   - Has `TaskCategory`-based recommendations from `Delegator`.
4. `src/debate.rs`: response synthesis and iteration use `from_prompt(&prompt, &self.cwd, None)`
   - Has full `Config` in context and can resolve backend defaults.
5. `src/backend/mod.rs::run_query_with_config`: public non-workflow helper passes `None` for all backend queries.
   - Has per-backend timeout and `Config`, so this is the right place to centralize context defaults.

## Approaches considered
### Approach A (selected): minimal mechanical propagation
- Add a local helper per module to derive a `StepContext` from available signals and pass it to `query(...)`.
- In `run_query_with_config`, compute model/timeout from `Config` per backend.
- Extend `Delegator::BackendProfile` with optional `model` where useful, and/or read selected backend config models where profiles are not available.
- Keep behavior changes localized to the 6 identified non-step call sites.

Pros: low refactor surface, straightforward migration, quick to implement and review.
Cons: repeated resolution logic across modules, possible semantic alignment questions for `Delegator` model ownership.

### Approach B: central context-factory abstraction
- Introduce a shared utility (e.g., `backend::context_from_execution_target`) that all non-step callers use.
- Add richer resolver traits for command/tool/team/debate contexts.

Pros: single point of truth and better long-term consistency.
Cons: broader API changes and higher refactor/testing surface for this small PR scope.

## Decision
Approach A is the best fit: it satisfies FR-20b with minimal churn and no broad API contract rewrite.

## Risks and assumptions
- `Config::defaults.model` is currently not present in this branch, so model defaults will need to be sourced from command-level overrides and backend config/model hints where available.
- `Team` paths currently have no explicit model signal from `Delegator` except optional new profile metadata; adding this field remains manageable and scoped.
- No external behavior change is expected for successful queries beyond context metadata.

## Planned implementation strategy
- Implement in order: `src/backend/mod.rs` (central helper for multi-backend calls), then `conductor`, `spawn`, `team`, `debate`.
- Keep `StepContext` history as `&[]` for all non-step callers unless upstream caller already has history.
- No new dependencies.
- Add targeted tests for one representative path per module where feasible.
