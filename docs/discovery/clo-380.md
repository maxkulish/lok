# Discovery Report: CLO-380 — FR-3b Codex `-o`/`--output-last-message` authoritative result extraction

**Task:** `FR-3b: Codex `-o`/`--output-last-message` authoritative result extraction`

**Date:** 2026-05-20

## Problem framing

Lok users who run Codex-backed workflow steps depend on the CLI output being parsed into reliable text and usage. Today, Codex success text is always derived from parsed JSONL events (`turn.completed` + `agent_message`), so if JSONL parsing is malformed the step can fail or lose the result even when Codex wrote a valid last-message file path in newer versions. Downstream validation and review flows then see brittle failures instead of recoverable results.

Who is affected: any workflow using `backend = "codex"` that expects robust final-answer extraction and token accounting, especially in CI and long-lived automation pipelines where output shape drifts are hardest to debug.

What we want: keep JSONL as the source for failure detection (`turn.failed` / `error`) and usage extraction, but use `-o / --output-last-message` when available as the authoritative success text path, with a safe fallback to the existing event parser only when the file is empty/absent.

Why now: PRD v5 already codifies FR-3b, FR-3a has landed and is in use, and this ticket is explicitly sequenced after it because the fallback and usage extraction rely on the event parser behavior.

## Existing code

### Codex backend query path

`src/backend/codex.rs` builds argv from configured args, injects sandbox + model, executes `codex` via `tokio::process::Command`, then parses `stdout` using `parse_output()`.

- `build_argv_prefix()` currently appends `--json`, `--ephemeral`-like defaults, `-s`, and `--model`.
- `query()` currently returns `super::codex_event::parse_jsonl_stream(&stdout)` for both success and failure paths, so JSONL parse health directly drives success/failure.

### Event parser

`src/backend/codex_event.rs` defines `CodexEvent` and `parse_jsonl_stream()`:

- Extracts final `agent_message` from the last `turn.completed` event.
- Extracts usage from `turn.completed`.
- Returns `BackendError::Parse` when no terminal completed turn is available.
- Returns `BackendError::ExecutionFailed` on `turn.failed`/`error`.

### Tests and fixtures

- `tests/codex_fixtures.rs` and `tests/codex_parse_output.rs` validate fixture shape and presence of error/non-error event types.
- `tests/fixtures/codex/` currently ships:
  - `turn-completed.jsonl`
  - `turn-failed.jsonl`
  - `multi-turn-reasoning.jsonl`
  - `missing-agent-message.jsonl`

No fixtures include Codex `-o` last-message companion files yet, and no code path currently passes `-o`.

### Baseline score

**Baseline: 4/10.** Event parsing is substantially improved from earlier substring logic and already feeds usage, but it still treats JSONL parsing as the sole result source. FR-3b requires orthogonal result extraction and deterministic fallback semantics that are currently missing.

## Approaches identified

### Approach 1 — Add temporary last-message file as authoritative success path (Recommended)

**Summary:** Allocate a per-query temp file via `tempfile`, pass `-o <path>` into argv when supported, read it after process exit, and only use JSONL text extraction when file is empty/absent.

**Pros**
- Aligns directly with FR-3b acceptance criteria.
- Keeps current parser/usage semantics for error and failure detection.
- Handles malformed JSONL on success by preserving the external last-message source.

**Cons**
- Requires careful handling of tempfile lifecycle and cleanup in all exit paths.
- Requires extra fixture coverage for non-empty, empty, and fail-with-`turn.failed` `-o` scenarios.

**Effort:** M

**Risk:** Medium (filesystem + cross-version flag behavior), mitigated by version gating or tolerant fallback.

### Approach 2 — Introduce a dual-channel parse result type before `query()` returns

**Summary:** Replace `parse_output()` with a richer extractor that always attempts to parse JSONL into `{agent_text?, usage?, terminal_error?}`, then let `query()` choose text source priority (`last-message` > JSONL text) and merge usage independently.

**Pros**
- Cleaner separation of concerns and explicit result composition.
- Makes failure/usage/text precedence semantics obvious and testable.
- Easier to extend for future structured extraction modes.

**Cons**
- Slightly larger refactor inside `codex.rs` and `codex_event.rs` return types.
- More unit/integration tests needed for the new composite result behavior.

**Effort:** M

**Risk:** Medium (behavioral surface changes around parse error propagation).

## Approach chosen

**Approach 1 — Add temporary last-message file as authoritative success path.**

This is the smallest and fastest path to satisfy FR-3b while preserving existing parser behavior for usage/failure handling. The code already has a stable event parser; using it only as secondary/fallback output source preserves the current success-path structure with minimal risk and clear acceptance-test mapping.
