# Discovery Report: CLO-395 — FR-12b opencode health probe + Google auth detection

## 1. Problem framing

**WHO:** Lok engines that warm up the `gemini` backend at startup (`warmup_backends`), the `lok doctor` command that displays backend health status (FR-14, CLO-393), and any automation that gates execution on backend availability.

**WHAT:** `GeminiBackend::health_check` (line 640 of `src/backend/gemini.rs`) is a dead stub — it returns `HealthStatus::new_available()` if `which::which("opencode")` succeeds, or `BackendError::Unavailable` otherwise. No version, no auth detection, no model enumeration, no 1s timeout budget. Users who configured opencode but failed `opencode auth login` see the backend as "available" and hit opaque failures later during query.

**WHY NOW:** CLO-394 migrated the Gemini backend to opencode subprocess invocation, making the deprecation path real. Users now need a health probe that catches misconfiguration before query time. This is the companion health probe to CLO-391 (Claude dual probe), CLO-392 (Codex probe), and CLO-389 (Ollama probe). Only the gemini backend lacks a real probe.

## 3. Approaches identified

### Approach A: Pure CLI probes (recommended)

Run three sequential `opencode` subcommands:
1. `opencode --version` (1s timeout) → `version`
2. `opencode auth list` (2s timeout) → parse for Google provider → `auth_method: "oauth" | "api-key"`, `mode: "oauth" | "api-key" | "none"`
3. `opencode models google` (3s timeout, optional) → `models: [ModelInfo]`

- **Pros:** matches opencode CLI contract directly, picks up any auth.json changes immediately, no filesystem layout assumptions beyond `~/.local/share/opencode/auth.json`.
- **Cons:** three subprocess spawns during warmup (serial, not parallel), auth list output is rich-text formatted (ANSI escapes in the table), models command is slow on first call.
- **Effort:** S
- **Risk:** Low

### Approach B: CLI probe + filesystem fallback

1. `opencode --version` (1s timeout) as primary probe.
2. Read `~/.local/share/opencode/auth.json` directly as JSON to detect auth_method — skip `opencode auth list` if file exists and parses.
3. Fall back to `opencode auth list` only if auth.json is missing or unparseable.
4. Environment check for `GEMINI_API_KEY` / `GOOGLE_API_KEY` as additional detection path.
5. Skip model enumeration by default (too slow); run on demand only.

- **Pros:** fastest warmup (one subprocess + one filesystem read), no rich-text parsing needed, env var check covers CI/container scenarios.
- **Cons:** auth.json path is platform-dependent (macOS/Linux share `~/.local/share/opencode/`; Windows uses `%LOCALAPPDATA%/opencode/`), filesystem coupling may break if opencode changes auth storage.
- **Effort:** S
- **Risk:** Low

### Approach C: Approach A + filesystem fallback (hybrid)

Combine the CLI-first approach with auth.json as backup when `opencode auth list` fails or times out.

- **Pros:** most resilient, handles both CLI and filesystem drift.
- **Cons:** slightly more code than Approach A alone.
- **Effort:** S-M
- **Risk:** Very Low

## 4. Chosen approach

**Approach C** — CLI probes with filesystem fallback.

**Rationale:**
- `opencode --version` is fast and reliable (verified at 1.15.10, returns plain text with ANSI).
- File-based auth detection (`~/.local/share/opencode/auth.json`) is the fastest path and matches the acceptance criteria ("read the auth.json file directly if `opencode auth list` is slow").
- Environment variable check (`GEMINI_API_KEY` / `GOOGLE_API_KEY`) handles CI/container environments where auth.json may not exist.
- Model enumeration is the slowest probe (~1-3s on first call) — we'll make it best-effort with a tight timeout, reporting in notes/notes only.
- Error notes must say `opencode auth login` (not `opencode login`) per the acceptance criteria.

### Probe order (all within budget)

1. **PATH check**: `which::which(self.command)` — immediate.
2. **Version**: `opencode --version` with **1s** tokio timeout → `HealthStatus.version`.
3. **Auth detection** (priority order):
   a. If version probe failed, skip auth detection → `mode: "none"`, diagnostic note.
   b. **File probe**: Read `~/.local/share/opencode/auth.json` (or `$HOME/.local/share/opencode/auth.json`). If the JSON contains a `"google"` key → `mode: "oauth"` (or `"api-key"` if `"type": "api"`), `auth_method: "oauth"`.
   c. **Env check**: If `GEMINI_API_KEY` or `GOOGLE_API_KEY` env var is set → `mode: "api-key"`, `auth_method: "api-key"`.
   d. **CLI probe**: `opencode auth list` with 2s timeout, grep output for "Google".
   e. None → `mode: "none"`, `available: false`, diagnostic note: "Run `opencode auth login` to set up Google credentials (not `opencode login`)."
4. **Model enumeration** (best-effort, 3s timeout): `opencode models google` → parse model names into `ModelInfo` list. On timeout or failure, leave `models: Vec::new()` and append a note to diagnostic.

### Key design decisions

- **Timeout budget**: Total worst-case ~6s (1s version + 2s auth + 3s models). This is acceptable for warmup which runs once at startup and is parallelized across all backends.
- **Serial probes**: The probes are sequential to respect the 1s version budget and avoid spawning 3 concurrent opencode subprocesses that could overload the system.
- **Available field**: `available: false` only when the binary is missing, times out, or no auth is detected. If the binary is present but `auth_method` is `"none"`, `available: false` with a diagnostic note prompting `opencode auth login`.
- **Mode discriminator**: `mode: "oauth"` for Google provider auth (opencode OAuth flow), `mode: "api-key"` for env var auth, `mode: "none"` for no auth. This matches `HealthStatus.mode` usage in Claude (`"api" | "cli"`).

## 6. Implementation outline (high level)

### `src/backend/gemini.rs`
- Replace the current `health_check()` stub with a multi-step probe function.
- Add private helper functions:
  - `probe_version(command: &str) -> Result<String, BackendError>` — spawns `--version` with 1s timeout, returns version string.
  - `probe_auth_file() -> Option<String>` — reads `~/.local/share/opencode/auth.json`, returns `"oauth"` if `google` key exists, `"api-key"` if `google.type == "api"`.
  - `probe_auth_env() -> Option<String>` — checks `GEMINI_API_KEY` / `GOOGLE_API_KEY` env vars.
  - `probe_auth_cli(command: &str) -> Result<Option<String>, BackendError>` — runs `opencode auth list` with 2s timeout, returns `"oauth"` if output contains "Google".
  - `probe_models(command: &str) -> Vec<ModelInfo>` — runs `opencode models google` with 3s timeout, parses model names.
- `health_check()` orchestrates the probes and returns `HealthStatus`.

### `src/main.rs`
- No changes needed (doctor check already references `opencode`).
